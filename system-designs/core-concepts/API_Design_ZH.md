# API Design

原文主题：系统设计面试中的 API 设计基础。

## 1. 常见 API 类型

面试里最常提到的 API 形态有三类：

1. REST
2. GraphQL
3. RPC

如果题目没有特别要求，REST 往往是最容易建立共识的默认答案。

## 2. 常见 API 模式

### 2.1 分页

分页是基础能力，大多数列表型接口都需要。

两种常见方式：

1. Offset-based pagination  
   例如：`/events?offset=20&limit=10`
2. Cursor-based pagination  
   例如：`/events?cursor=c989&limit=10`

经验上：

1. 数据规模小、随机跳页需求明显时，可以先用 offset。
2. 实时数据流、大表深翻页或需要稳定翻页时，更适合 cursor。

### 2.2 版本管理

API 会演进，必须考虑如何在不打挂旧客户端的情况下持续升级。

常见做法有两种：

1. URL Versioning  
   例如：`/v1/events`、`/v2/events`
2. Header Versioning  
   例如：`Accept-Version: v2`

面试里默认优先推荐 URL 版本号，原因是：

1. 更直观
2. 更容易讲清楚
3. 工程上更常见

除非面试官特意追问，否则不用主动展开 Header 版本管理。

## 3. 安全设计

API 安全不需要讲成完整安全体系，但你必须体现出“这是生产系统”的意识。

至少要覆盖：

1. 认证
2. 授权
3. 限流

### 3.1 Authentication vs Authorization

认证回答的是“你是谁”，授权回答的是“你能做什么”。

完整检查顺序通常是：

1. 用户是否已登录或已验证身份
2. 用户是否有权限执行当前操作

### 3.2 API Key vs JWT

#### API Key

更适合：

1. 内部服务调用
2. 给第三方开发者开放接口

不适合直接做用户态登录会话，因为：

1. 用户不适合直接管理长字符串凭证
2. API Key 通常不天然携带用户上下文
3. 过期和权限粒度控制不如用户会话机制自然

#### JWT

更适合：

1. Web 用户会话
2. 移动端用户会话

优点：

1. 可以携带用户 ID、权限和过期时间
2. 服务端校验签名后即可识别用户身份
3. 在分布式系统里，多个服务都可以独立校验

面试里的稳妥说法是：

1. 用户会话优先 JWT 或 session cookie
2. 服务间或开发者接口优先 API Key

## 4. RBAC

真实系统里通常有多种角色，例如：

1. 普通用户
2. 运营/管理员
3. 商家/场馆经理

RBAC 的核心思想是：

1. 给用户分配角色
2. 给角色分配权限
3. 接口先做认证，再按角色做授权

例如：

```text
GET /bookings/{id}
1. 用户是否已认证
2. 用户是否是该订单拥有者，或者拥有管理员权限
```

## 5. Rate Limiting

限流的目标是防止滥用、攻击和误用。

常见维度包括：

1. 按用户限流
2. 按 IP 限流
3. 按接口维度限流

触发限流后，通常返回 `429 Too Many Requests`。

限流逻辑一般放在：

1. API Gateway
2. 应用中间件

## 6. 面试怎么讲 API Design

系统设计面试里的 API 设计，核心不是写 OpenAPI 文档，而是证明你有工程判断。

推荐表达顺序：

1. 先选 API 风格
2. 再定义核心资源和关键接口
3. 补分页、版本管理和认证方式
4. 最后说明授权和限流

只要这条链路清晰，通常就足够了。

## 补充内容

Notion 原文已经覆盖分页、版本和安全，我补了一层更适合面试复述的结构：

1. 资源建模：围绕实体设计接口，而不是围绕页面按钮设计。
2. 协议风格：REST 作为默认，复杂查询再考虑 GraphQL，内部高性能链路再谈 RPC/gRPC。
3. 请求语义：幂等接口、非幂等接口、异步接口分别如何设计。
4. 兼容性：版本演进、废弃策略、默认字段扩展方式。
5. 安全性：认证、授权、限流、审计。
6. 可用性：超时、重试、错误码设计、幂等键。

## 常见问题

1. 系统设计面试里什么时候应该默认选 REST？
- 当题目没有强调灵活查询、流式交互或内部高性能 RPC 时，默认选 REST。
- 它最容易讲清楚，也最容易和面试官达成共识。

2. Offset 分页和 Cursor 分页分别适合什么场景？
- Offset 适合中小数据量和随机跳页。
- Cursor 适合大表、实时流和需要稳定翻页的场景。

3. API 为什么需要版本管理？
- 因为接口一定会演进，而客户端不会同时升级。
- 版本管理的本质是兼容老调用方。

4. Header 版本和 URL 版本该怎么取舍？
- 面试里默认说 URL 版本最稳。
- Header 版本更“规范”，但表达和调试都不如 URL 直接。

5. JWT 和 API Key 分别适合哪些场景？
- JWT 更适合用户会话。
- API Key 更适合服务间调用或开放平台场景。

6. 认证和授权为什么必须分开讲？
- 因为“你是谁”和“你能做什么”是两件事。
- 用户登录成功，不代表他有权限访问所有资源。

7. RBAC 适合解决什么问题？
- 适合多角色、多权限系统。
- 它把权限管理从“按用户散配”收敛成“按角色配置”。

8. 为什么限流通常放在 API Gateway 或中间件层？
- 因为这是最靠前、最统一的拦截点。
- 可以在业务逻辑之前就挡住滥用流量。

9. 幂等键适合放在哪些创建类接口中？
- 下单、支付、预约、登录确认这类“重复提交代价高”的接口都适合。
- 原则是：重复请求不能产生重复副作用。

10. 面试里 API 设计讲到什么深度就够了？
- 讲到资源建模、关键接口、分页、认证授权、限流，通常就够了。
- 除非面试官追问，不需要把 API 设计讲成完整规范文档。
