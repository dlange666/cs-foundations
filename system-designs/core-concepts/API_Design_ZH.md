# API Design

原文主题：系统设计面试中的 API 设计基础。

## 1. 常见 API 类型

面试里最常提到的 API 形态有三类：

1. REST
2. GraphQL
3. RPC

如果题目没有特别要求，REST 往往是最容易建立共识的默认答案。

## 2. 常见 API 模式

### 2.1 分页

分页是基础能力，大多数列表型接口都需要。

两种常见方式：

1. Offset-based pagination  
   例如：`/events?offset=20&limit=10`
2. Cursor-based pagination  
   例如：`/events?cursor=c989&limit=10`

经验上：

1. 数据规模小、随机跳页需求明显时，可以先用 offset。
2. 实时数据流、大表深翻页或需要稳定翻页时，更适合 cursor。

### 2.2 版本管理

API 会演进，必须考虑如何在不打挂旧客户端的情况下持续升级。

常见做法有两种：

1. URL Versioning  
   例如：`/v1/events`、`/v2/events`
2. Header Versioning  
   例如：`Accept-Version: v2`

面试里默认优先推荐 URL 版本号，原因是：

1. 更直观
2. 更容易讲清楚
3. 工程上更常见

除非面试官特意追问，否则不用主动展开 Header 版本管理。

## 3. 安全设计

API 安全不需要讲成完整安全体系，但你必须体现出“这是生产系统”的意识。

至少要覆盖：

1. 认证
2. 授权
3. 限流

### 3.1 Authentication vs Authorization

认证回答的是“你是谁”，授权回答的是“你能做什么”。

完整检查顺序通常是：

1. 用户是否已登录或已验证身份
2. 用户是否有权限执行当前操作

### 3.2 API Key vs JWT

#### API Key

更适合：

1. 内部服务调用
2. 给第三方开发者开放接口

不适合直接做用户态登录会话，因为：

1. 用户不适合直接管理长字符串凭证
2. API Key 通常不天然携带用户上下文
3. 过期和权限粒度控制不如用户会话机制自然

#### JWT

更适合：

1. Web 用户会话
2. 移动端用户会话

优点：

1. 可以携带用户 ID、权限和过期时间
2. 服务端校验签名后即可识别用户身份
3. 在分布式系统里，多个服务都可以独立校验

面试里的稳妥说法是：

1. 用户会话优先 JWT 或 session cookie
2. 服务间或开发者接口优先 API Key

## 4. RBAC

真实系统里通常有多种角色，例如：

1. 普通用户
2. 运营/管理员
3. 商家/场馆经理

RBAC 的核心思想是：

1. 给用户分配角色
2. 给角色分配权限
3. 接口先做认证，再按角色做授权

例如：

```text
GET /bookings/{id}
1. 用户是否已认证
2. 用户是否是该订单拥有者，或者拥有管理员权限
```

## 5. Rate Limiting

限流的目标是防止滥用、攻击和误用。

常见维度包括：

1. 按用户限流
2. 按 IP 限流
3. 按接口维度限流

触发限流后，通常返回 `429 Too Many Requests`。

限流逻辑一般放在：

1. API Gateway
2. 应用中间件

## 6. 面试怎么讲 API Design

系统设计面试里的 API 设计，核心不是写 OpenAPI 文档，而是证明你有工程判断。

推荐表达顺序：

1. 先选 API 风格
2. 再定义核心资源和关键接口
3. 补分页、版本管理和认证方式
4. 最后说明授权和限流

只要这条链路清晰，通常就足够了。

## 补充内容

Notion 原文已经覆盖分页、版本和安全，我补了一层更适合面试复述的结构：

1. 资源建模：围绕实体设计接口，而不是围绕页面按钮设计。
2. 协议风格：REST 作为默认，复杂查询再考虑 GraphQL，内部高性能链路再谈 RPC/gRPC。
3. 请求语义：幂等接口、非幂等接口、异步接口分别如何设计。
4. 兼容性：版本演进、废弃策略、默认字段扩展方式。
5. 安全性：认证、授权、限流、审计。
6. 可用性：超时、重试、错误码设计、幂等键。
