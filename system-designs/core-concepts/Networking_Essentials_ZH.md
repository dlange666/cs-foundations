# Networking Essentials（中文改写）

原文主题：Networking Essentials（系统设计核心概念）

## 学习目标

这篇内容的核心是：在系统设计面试中，你需要掌握“够用且可落地”的网络知识，而不是陷入纯底层细节。重点是能解释清楚协议选择、负载均衡、跨地域延迟与故障处理的取舍。

## 1. Networking 101

### 1.1 网络分层的面试视角

面试不要求你背完整协议栈细节，但你需要知道：
1. 不同层解决的问题不同。
2. 你的系统设计决策会落在不同层（L4/L7、TCP/UDP、HTTP/WebSocket 等）。

### 1.2 一个简单 Web 请求的链路

典型流程：
1. DNS 解析域名到 IP。
2. TCP 三次握手建立连接（SYN -> SYN-ACK -> ACK）。
3. 客户端发 HTTP 请求。
4. 服务端处理并返回响应。
5. TCP 四次挥手关闭连接（FIN/ACK）。

面试里经常忽略的点：很多请求延迟并不只来自业务代码，还来自网络握手、跨地域链路、队列拥塞等。

## 2. 传输层协议：UDP vs TCP

### 2.1 UDP：快，但不保证可靠

特征：
1. 无连接。
2. 不保证送达。
3. 不保证顺序。
4. 开销小、延迟低。

适用场景：
1. 实时性优先（语音、视频、游戏）。
2. 少量丢包可接受。
3. 高频遥测/日志等场景。

### 2.2 TCP：可靠，但有额外开销

特征：
1. 面向连接。
2. 可靠传输（顺序、重传、校验）。
3. 流量控制。
4. 拥塞控制。

面试默认策略：不确定时优先 TCP（尤其是业务 API 场景）。

## 3. 应用层协议选型

## 3.1 HTTP/HTTPS

你要会讲：
1. 方法语义：GET/POST/PUT/PATCH/DELETE。
2. 状态码分层：2xx、3xx、4xx、5xx。
3. Header/Body 的职责。

## 3.2 REST

优势：
1. 简单、成熟、生态好。
2. 面试里可作为默认 API 风格。

劣势：
1. 对复杂数据聚合场景，可能出现过取/欠取。
2. JSON 序列化相对二进制协议更重。

一句话：面试中没特殊要求时，REST 是稳定起点。

## 3.3 GraphQL

解决的问题：前端按需取数，减少过取/欠取。

适用：
1. 前端快速迭代。
2. 客户端对返回字段灵活性要求高。

代价：
1. 后端执行与治理复杂度提升。
2. 查询性能治理（N+1、复杂查询限流）需要额外投入。

面试建议：只有在“灵活查询”是明确核心需求时再选。

## 3.4 gRPC

优势：
1. Protobuf 二进制协议，序列化开销小。
2. HTTP/2，多路复用，服务间通信性能好。
3. 强类型接口定义，跨语言协作清晰。

适用：
1. 内部微服务调用。
2. 高吞吐低延迟链路。

不适合：公网浏览器直连场景（生态与兼容性限制较大）。

## 3.5 SSE（Server-Sent Events）

特点：
1. 服务端单向推送。
2. 基于 HTTP，落地相对简单。

适用：
1. 通知、订阅更新、轻量实时推送。

局限：
1. 连接保活与重连逻辑要处理。
2. 中间网络设备可能影响流式行为。

## 3.6 WebSocket

特点：
1. 全双工实时通信。
2. 协议升级后保持长连接。

适用：
1. 聊天、协同编辑、实时互动。

代价：
1. 连接是有状态的，对负载均衡和扩缩容要求更高。
2. 基础设施（代理、网关、LB）需确认支持。

## 3.7 WebRTC

定位：浏览器端点对点通信，常用于音视频。

关键组件：
1. 信令服务器（交换会话信息）。
2. STUN（NAT 穿透）。
3. TURN（中继兜底）。

面试建议：除非题目就是音视频/会议系统，否则不要过早引入 WebRTC 复杂度。

## 4. 负载均衡（Load Balancing）

### 4.1 L4 与 L7

L4（传输层）：
1. 关注连接分发。
2. 性能高、决策简单。
3. 适合连接型场景（如大量 WebSocket 连接）。

L7（应用层）：
1. 能按 URL/Header/Cookie 路由。
2. 功能强，但开销更高。
3. 适合 HTTP 业务路由与网关策略。

### 4.2 常见调度算法

1. Round Robin
2. Random
3. Least Connections
4. Least Response Time
5. IP Hash

实战建议：
1. 无状态服务常用 Random / Round Robin。
2. 长连接场景常用 Least Connections。

### 4.3 健康检查与高可用

LB 会通过健康检查摘除异常节点并自动恢复流量，常见检查方式：
1. TCP 探活。
2. HTTP 探活（如 `/health` 返回 200）。

## 5. 常见深挖：跨地域与故障处理

### 5.1 跨地域与延迟

核心事实：物理距离直接决定网络时延下界（光速限制无法绕开）。

优化方向：
1. 数据就近（data locality）。
2. CDN 缓存静态/可缓存动态内容。
3. 区域化分片（regional partitioning），让请求尽量在本地闭环。

### 5.2 超时、重试、退避

故障是常态，网络调用可能超时、失败、抖动。

标准做法：
1. 设置超时。
2. 重试 + 指数退避（exponential backoff）。
3. 加抖动（jitter），避免重试风暴同步打爆系统。

### 5.3 幂等性（Idempotency）

重试有副作用时（如支付扣款），必须保证幂等。

常见手段：
1. 幂等键（idempotency key）。
2. 服务端去重与状态机控制。
3. 重复请求返回同一业务结果。

### 5.4 熔断器（Circuit Breaker）

作用：防止依赖故障引发级联雪崩。

典型状态：
1. Closed：正常放行。
2. Open：快速失败，不再压垮下游。
3. Half-Open：探测恢复后再关闭。

收益：
1. Fail Fast。
2. 保护下游。
3. 提升整体稳定性。

## 6. 面试中的默认答题策略

1. 默认外部 API 用 REST/HTTPS。
2. 内部高性能调用可选 gRPC。
3. 需要实时推送先考虑 SSE，强双向再上 WebSocket。
4. HTTP 业务优先 L7 LB，长连接强调 L4/LB 支持策略。
5. 全链路都要补：超时、重试、幂等、熔断。
6. 涉及全球用户时优先谈：CDN + 区域化部署 + 数据就近。

## 7. 总结

Networking 在系统设计里的价值不是“协议背诵”，而是“能把性能、可靠性、复杂度三者做可解释取舍”。

你只要能稳定回答下面三件事，网络这一块就够面试：
1. 为什么选这个协议？
2. 在高并发和故障下会发生什么？
3. 你如何让系统在现实网络里依然可用？
