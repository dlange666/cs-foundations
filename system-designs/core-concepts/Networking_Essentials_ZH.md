# Networking Essentials

原文主题：系统设计面试中的网络基础。

## 学习目标

系统设计面试里不要求你讲底层网络实现细节，但你需要会做协议选择、负载均衡选择，以及跨地域延迟与故障处理的取舍。

## 1. 网络分层的面试视角

对应用工程师来说，最常出现的是三层：

1. L3：Network Layer
2. L4：Transport Layer
3. L7：Application Layer

面试不需要背完整 OSI 七层，但要知道：

1. IP 负责寻址与路由
2. TCP / UDP / QUIC 负责端到端传输
3. HTTP / WebSocket / DNS 这些协议位于应用层

## 2. 一个请求是怎么走的

一个普通 Web 请求通常包含：

1. DNS 解析域名
2. 建立 TCP 连接
3. 发送 HTTP 请求
4. 服务端返回结果
5. 关闭或复用连接

很多响应慢的问题并不只来自业务代码，还会受到握手、跨地域链路和网络抖动影响。

## 3. UDP vs TCP

### UDP

特点：

1. 无连接
2. 不保证送达
3. 不保证顺序
4. 延迟低、开销小

适用：

1. 语音视频
2. 游戏
3. 对少量丢包可容忍的实时场景

### TCP

特点：

1. 面向连接
2. 可靠传输
3. 有顺序保证
4. 有流量控制和拥塞控制

大多数业务 API 默认选 TCP。

## 4. 应用层协议选择

### HTTP / HTTPS

这是系统设计里最常见的默认协议组合，适用于绝大多数外部 API。

### REST

优势：

1. 简单
2. 工程实践成熟
3. 易于和面试官对齐

大多数题目没有特殊需求时，REST 是最稳妥的默认选择。

### GraphQL

优点是灵活取数，缺点是服务端执行和治理成本更高。只有在前端灵活查询是核心需求时才值得引入。

### gRPC

适合内部服务间高性能通信，优点是：

1. 二进制协议
2. HTTP/2
3. 强类型接口定义

它不适合直接作为浏览器公网 API 的默认选择。

### SSE

SSE 是服务端到客户端的单向推送，适合通知流、轻量实时更新。

### WebSocket

WebSocket 支持双向实时通信，适合聊天、协同编辑、实时互动。

代价是连接有状态，对网关和负载均衡要求更高。

### WebRTC

这更偏音视频和浏览器点对点通信场景。除非题目就是会议或音视频，否则不要过早引入。

## 5. 负载均衡

### L4

按连接分发，性能高，适合长连接或较基础的连接层转发。

### L7

按 HTTP 语义做路由，能基于 URL、Header、Cookie 分流，更灵活。

面试里如果是 HTTP 服务，优先谈 L7；如果是大量长连接，也要补充 L4/LB 支持策略。

## 6. 跨地域与故障处理

### 跨地域延迟

物理距离决定了延迟下界。优化思路通常包括：

1. 数据就近
2. CDN
3. 区域化部署

### 超时与重试

标准做法：

1. 设置超时
2. 指数退避重试
3. 加 jitter，避免重试风暴

### 幂等

涉及支付、创建订单等操作时，必须考虑幂等。

常见方法：

1. 幂等键
2. 去重记录
3. 状态机约束

### 熔断

当下游服务异常时，要快速失败，避免把问题放大成级联雪崩。

## 7. 面试默认答题策略

1. 外部 API 默认 REST + HTTPS
2. 内部高性能链路可以补 gRPC
3. 轻量实时更新优先 SSE
4. 双向强实时才引入 WebSocket
5. 所有网络调用都补上超时、重试、幂等、熔断

## 补充内容

Notion 原文覆盖了常见协议和负载均衡，但还缺少一层“网络设计决策树”，我补成下面这套结构：

1. 连接类型：短连接还是长连接。
2. 传输选择：TCP、UDP、QUIC 分别适合什么场景。
3. 应用协议：REST、gRPC、SSE、WebSocket 怎么选。
4. 网关与负载均衡：L4 还是 L7，是否需要 sticky session。
5. 跨地域问题：DNS、CDN、区域路由、就近访问。
6. 失败处理：超时、重试、jitter、熔断、幂等。
7. 安全基础：TLS、证书、认证头、零信任边界。
