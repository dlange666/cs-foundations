# Data Modeling

原文主题：系统设计面试中的数据建模基础。

## 1. 数据库模型选型

常见数据库模型包括：

1. Relational Database（SQL）
2. Document Database
3. Key-Value Store
4. Wide-Column Database
5. Graph Database

面试里最重要的不是把所有模型展开，而是根据访问模式选最合理的那种。

### Graph Database 的提醒

图数据库在面试里经常被误用。它听起来高级，但很多题目根本不需要它。

即便是关系图很强的公司，核心链路也未必依赖图数据库。多数时候，关系型数据库或其他常见存储已经足够支撑主查询模式。

## 2. Schema Design Fundamentals

### 2.1 从需求出发

数据建模前先看三件事：

1. 数据量
2. 访问模式
3. 一致性要求

如果这三点没讲清楚，后面的表结构讨论容易失焦。

### 2.2 Entities, Keys, Relationships

先识别核心实体，再确定：

1. 主键
2. 外键或关联方式
3. 一对一、一对多、多对多关系

这一步是高层设计和数据库选型之间的桥梁。

### 2.3 为访问模式建索引

索引不是“看到字段就加”，而是围绕查询模式设计。

你应该能回答：

1. 系统最常见的查询是什么
2. 哪些字段会被频繁过滤、排序或关联
3. 哪些索引能真正提升关键路径性能

### 2.4 Normalize vs Denormalize

规范化有助于减少冗余、保证更新一致性；反规范化则常用于读性能优化。

面试里稳妥的说法是：

1. 先给出清晰、可维护的基础模型
2. 当读路径明显成为瓶颈时，再有针对性地反规范化

## 3. Scaling and Sharding

如果系统大到单库撑不住，就要考虑分片。

原文强调的重点是：分片键必须和主访问模式对齐。

例如，如果主要查询是“某个用户的帖子”，那 `user_id` 往往是比时间更合理的分片键，因为它能把同一用户的数据放在一起，减少跨分片查询。

### 时间分片的风险

时间范围分片看起来适合“最近数据”场景，但如果系统写入很重，所有新写入都会集中打到最新分片，容易形成热点。

因此：

1. 时间分片更适合归档、历史分析类场景
2. 对高写入在线系统要谨慎使用

### Cross-Shard Query 的代价

如果分片键和查询模式不一致，就会出现跨分片查询。

例如：

1. 你按 `user_id` 分片
2. 但时间线要聚合多个关注对象的数据
3. 那你就需要同时打多个分片再做合并

这会显著增加系统复杂度和延迟。

### 分片键是长期决策

分片键一旦定错，后续代价会很高。它会影响：

1. 查询成本
2. 数据迁移成本
3. 热点分布
4. 后续扩缩容方式

## 4. 面试中的落点

数据建模是系统设计的重要组成部分，但通常不是整场面试的唯一重点。

推荐回答顺序：

1. 先列核心实体
2. 选数据库类型
3. 确定字段、主键和关系
4. 为关键访问路径补索引
5. 评估是否需要反规范化
6. 再判断是否需要分片

只要你能围绕访问模式解释这些取舍，就足够体现工程判断。

## 补充内容

Notion 原文主要讲了数据库模型和分片注意点，我补了一套更完整的数据建模检查结构：

1. 实体识别：核心对象、生命周期、所有权。
2. 查询模式：按主键查、按时间查、按关联查、按聚合查。
3. 索引策略：主索引、二级索引、联合索引、覆盖索引。
4. 一致性要求：强一致、最终一致、读写隔离要求。
5. 规范化与反规范化：什么时候为了读性能复制字段。
6. 扩展策略：冷热分层、分区、分片、归档。

## 常见问题

1. 数据建模时为什么要先看访问模式？
- 因为表结构、索引和分片都是围绕“怎么查”设计的。
- 不先看访问模式，建模很容易漂亮但不可用。

2. SQL 和 NoSQL 的取舍核心是什么？
- 不是谁先进，而是谁更匹配访问模式和一致性要求。
- 半逐字稿：`如果关系复杂、事务重要，我更偏 SQL；如果模型灵活、扩展优先，我会考虑 NoSQL。`

3. 什么情况下应该反规范化？
- 当读性能是核心瓶颈，且接受一定冗余和同步成本时。
- 常见于 Feed、聚合视图、搜索展示页。

4. 索引为什么不能“想当然全加”？
- 索引会占空间，也会拖慢写入。
- 只应该为高频且关键的查询模式建索引。

5. 图数据库在面试里为什么经常被误用？
- 因为很多关系场景用 SQL 就能解决。
- 面试里盲目上图数据库，通常说明没有先分析主查询路径。

6. 时间分片为什么容易产生热点？
- 因为最新写入会集中打到最新分片。
- 写流量重的在线系统里，这通常是风险点。

7. 跨分片查询为什么昂贵？
- 因为要打多个分片、聚合多个结果，还要承受更高尾延迟。
- 它的代价不只是慢，还包括系统复杂度上升。

8. 分片键为什么会成为长期架构约束？
- 因为它决定数据如何分布，也决定查询成本。
- 后期重选 shard key 往往意味着大规模迁移。

9. 面试里数据建模应该讲到字段级还是实体级？
- 默认先讲实体级和关键字段。
- 只有当字段本身影响索引、约束或查询路径时，再展开到字段级。

10. 什么时候需要从单库过渡到分区或分片？
- 当单库在容量、读写吞吐或维护窗口上明显撑不住时。
- 在那之前，优先用索引、缓存、读写分离等低成本手段。
