# Consistent Hashing

原文主题：一致性哈希如何降低扩缩容时的数据迁移成本。

## 1. 为什么需要一致性哈希

假设你把数据通过下面的方式分配到 3 台数据库：

```text
database_id = hash(event_id) % 3
```

这个方案在节点数量固定时能工作，但一旦从 3 台扩到 4 台，大量 key 的映射都会改变，结果就是几乎整库数据都要重新搬迁。

这就是简单取模分片的问题。

## 2. 一致性哈希的核心思想

一致性哈希把两类对象都放到一个哈希环上：

1. 数据 Key
2. 节点

查找方式是：

1. 先对 key 做哈希
2. 在环上找到它的位置
3. 顺时针找到遇到的第一个节点

这样做的结果是：当节点增加或减少时，只会影响局部区间，而不是全量重排。

## 3. 节点变更时发生什么

### 新增节点

如果新增一个节点，只需要把它前一个节点和它之间那一小段区间的数据迁移给它。

换句话说：

1. 只有局部数据需要移动
2. 其余 key 的归属保持不变

### 节点下线

如果一个节点故障，只需要把原来落在这个节点上的 key 顺时针交给后继节点。

这同样是局部影响，而不是全局洗牌。

## 4. Virtual Nodes

如果每个物理节点在哈希环上只占一个点，数据分布可能不均衡。

解决方法是给每个物理节点映射多个虚拟节点。

这样做的收益：

1. 数据分布更均匀
2. 节点上下线时负载转移更平滑

虚拟节点解决的是“结构性不均衡”。

## 5. 热点问题

即使数据分布均匀，访问流量也可能不均匀。

例如：

1. 某个演唱会 Key 极热
2. 某个明星用户流量远高于普通用户

一致性哈希本身只能均匀分布 key，不能均匀分布热点流量。

常见解决方式：

1. Read Replicas
2. Key Salting
3. 自适应重平衡

面试里要把这句话说清楚：

1. Virtual nodes 解决结构性负载不均
2. Replication / salting 解决业务热点不均

## 6. 数据迁移的工程现实

一致性哈希只告诉你“数据应该放在哪里”，并不会自动完成 TB 级数据瞬间迁移。

实际生产系统里，通常还会结合：

1. 数据复制
2. 后台搬迁
3. 故障转移

来保证节点变动期间系统仍可用。

## 7. 现实中的应用

一致性哈希常见于：

1. Cassandra
2. Dynamo 风格系统
3. CDN
4. 分布式缓存

不过要注意，工程里“consistent hashing”这个词有时会被泛化使用。有些系统使用的是固定 slot 或其他变体，但核心目标仍然是：节点变化时尽量少搬数据。

## 8. 面试里什么时候讲一致性哈希

如果你用的是 DynamoDB、Cassandra 这类现成分布式系统，通常只需要说明它们底层会处理数据分布与扩容。

真正需要深入展开一致性哈希的场景通常是：

1. 设计分布式数据库
2. 设计分布式缓存
3. 设计分布式消息系统

也就是偏底层基础设施题。

## 9. 结论

一致性哈希解决的是一个非常关键的问题：

当节点数量变化时，如何把数据重新分配，但尽量少搬数据。

面试里你至少要能讲清楚三点：

1. 为什么它比简单取模更好
2. 为什么需要虚拟节点
3. 为什么它仍然不能单独解决热点流量问题

## 补充内容

Notion 原文集中在哈希环和虚拟节点，我补了一套完整理解框架：

1. 基线问题：简单取模扩容时会导致大规模重映射。
2. 环模型：节点和 key 共同映射到一个逻辑环。
3. 局部迁移：节点变动只影响相邻区间。
4. 虚拟节点：缓解结构性负载不均。
5. 热点治理：副本、key salting、自适应迁移。
6. 工程现实：一致性哈希通常与复制、故障转移和后台搬迁一起使用。

## 常见问题

1. 一致性哈希解决的核心问题是什么？
- 核心是节点变化时尽量少搬数据。
- 它解决的是扩容、缩容和节点故障时的重映射成本。

2. 为什么简单取模在扩容时代价很高？
- 因为节点数一变，几乎所有 key 的映射都可能变。
- 这会导致大规模数据迁移和缓存失效。

3. 哈希环是如何决定某个 key 落到哪个节点上的？
- 先对 key 做哈希，再在环上顺时针找到第一个节点。
- 这个节点就是 key 的归属节点。

4. 为什么需要虚拟节点？
- 因为一个物理节点只占一个点时，分布可能不均衡。
- 虚拟节点让每台机器在环上出现多次，分布更平滑。

5. 虚拟节点能否解决热点流量问题？
- 不能完全解决。
- 它解决的是结构性不均衡，不解决某些 key 天然更热的问题。

6. 节点下线时数据会发生什么？
- 原来属于该节点的区间会转移给后继节点。
- 影响的是局部区间，不是全量 key。

7. 节点新增时为什么只需要迁移局部数据？
- 因为新节点只接管它所在区间前一段的数据。
- 其他区间映射不变，所以迁移范围有限。

8. 一致性哈希和复制通常为什么要一起出现？
- 因为哈希只解决“放哪”，复制才解决容错和高可用。
- 真实系统通常不会只靠哈希分布而不做副本。

9. Redis Cluster 为什么不属于经典哈希环实现？
- 因为它用的是固定 hash slot，而不是 textbook 的环。
- 但它的目标仍然类似：让扩缩容时的数据迁移更可控。

10. 面试里什么时候要深入讲一致性哈希？
- 当题目是分布式缓存、分布式数据库或消息系统时。
- 如果你用的是现成中间件，通常点到为止就够了。
