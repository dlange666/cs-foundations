# System Design - In a Hurry

原文主题：快速建立系统设计面试的答题框架。

## 1. 先打基础

系统设计面试的准备顺序不是先刷题，而是先建立一个稳定的思考框架。

建议先完成三件事：

1. 理解系统设计面试到底在考什么。
2. 固定一套自己的答题顺序。
3. 先补核心概念、关键技术和常见模式。

这些基础内容看起来偏抽象，但它们决定了你后续在题目里能不能快速做出合理取舍。

## 2. 练习方式

真正有效的准备不是只看答案，而是反复做完整推演。

推荐节奏：

1. 选一道题。
2. 先自己独立回答。
3. 再看参考答案。
4. 对照差距补知识点。
5. 过几天重新答一遍。

目标不是背模板，而是把框架练成默认反应。

## 3. 面试交付框架

系统设计面试里最常见的失败方式，不是不会，而是最后没交付出一个可工作的系统。

一个稳定的答题框架通常包括：

1. 需求澄清
2. 核心实体
3. API 或系统接口
4. 数据流
5. 高层架构
6. 深入讨论

### 3.1 需求澄清

先确认功能需求和非功能需求。

功能需求回答的是“系统要做什么”；非功能需求回答的是“系统做到什么程度才算合格”。

非功能需求至少应覆盖：

1. 延迟目标
2. 吞吐规模
3. 一致性与可用性取舍
4. 持久性要求
5. 容错能力
6. 安全与合规

### 3.2 容量估算

不要机械地一开始就做大段估算，但当某个架构决策依赖数量级时，必须补估算。

例如：

1. 单库是否还能扛住读写流量
2. 是否需要缓存
3. 是否需要分片
4. 是否必须引入消息队列

### 3.3 核心实体

先列出最关键的数据对象，不需要一开始就把数据表设计到字段级。

核心实体的作用是：

1. 统一术语
2. 锚定系统骨架
3. 为后续 API 和存储设计做准备

### 3.4 API / 系统接口

面试里通常选择 REST、GraphQL 或 RPC 中最合适的一种。

默认情况下，REST 是最稳的起点。接口设计应当和前面确认过的功能需求直接对应。

### 3.5 数据流

对有明显处理链路的系统，要明确描述：

`请求输入 -> 核心处理 -> 数据读写 -> 结果输出`

这一步可以帮助你避免只是在白板上堆组件。

### 3.6 高层设计

先交付最小可行架构，再逐步扩展复杂度。

这一步通常包含：

1. 客户端
2. API 层
3. 应用服务
4. 数据库
5. 缓存
6. 异步链路

错误做法是太早展开复杂优化，导致主线没有讲完。

### 3.7 深入讨论

当高层架构成立后，再进入深挖：

1. 性能瓶颈
2. 热点问题
3. 一致性问题
4. 可扩展性
5. 容灾与恢复
6. 安全问题

高级别面试通常会更看重你能否主动发起这部分讨论。

## 4. 核心概念

这部分是系统设计的通用语言。常见知识块包括：

1. Networking
2. API Design
3. Data Modeling
4. Caching
5. Sharding
6. Consistent Hashing
7. CAP

这些概念不是可选项，几乎所有题都会反复用到。

## 5. 关键技术

面试准备不需要把所有技术都学全，但需要对高频技术形成稳定认知，例如：

1. Redis
2. Kafka
3. PostgreSQL
4. DynamoDB
5. Cassandra
6. API Gateway

重点不是把产品文档背下来，而是知道：

1. 这个技术解决什么问题
2. 什么时候用
3. 什么时候不用
4. 它带来的代价是什么

## 6. 常见模式

系统设计里经常反复出现的模式包括：

1. 实时更新
2. 读扩展
3. 写扩展
4. 多步骤流程
5. 长任务管理
6. 大对象处理
7. 并发争用处理

你不需要先把所有模式讲全，但要在题目里识别出它们何时出现。

## 7. 这篇内容对准备的价值

这篇文章的价值主要在于帮你搭建“答题框架”，而不是替代深入的分布式系统学习。

更准确的使用方式是：

1. 用它固定回答结构
2. 用具体题目练交付
3. 用 DDIA、工程案例和技术文档补深度

## 补充内容

Notion 原始内容更偏答题框架，我补了一层更适合长期准备的知识结构：

1. 面试交付层：需求澄清、容量估算、API、数据流、高层设计、deep dive。
2. 基础概念层：网络、缓存、分片、一致性、异步处理、数据建模。
3. 技术组件层：数据库、缓存、消息队列、搜索、对象存储、网关。
4. 模式层：读扩展、写扩展、实时更新、长任务、热点处理、多步骤流程。
5. 演练层：把固定框架套到 3-5 道高频题上，练 30 分钟完整口述。
6. 复盘层：每道题补“瓶颈、容灾、一致性、安全、扩展”五个追问。

## 常见问题

1. 系统设计面试的标准答题顺序是什么？
2. 需求澄清时应该优先问哪些问题？
3. 容量估算什么时候必须做，什么时候可以跳过？
4. 高层设计和 deep dive 的时间应该怎么分配？
5. 面试里什么时候要主动引入缓存或分片？
6. 如何避免“讲了很多技术但没有交付完整方案”？
7. 如何在一致性、可用性和延迟之间做取舍表达？
8. 面试官追问失败场景时，应该从哪些维度回答？
9. 如何把一个题从“会做”练到“能口述清楚”？
10. 准备系统设计时，框架学习和原理学习的比例应该怎么安排？
