# Redis

原文主题：为什么 Redis 是系统设计面试里最值得深入掌握的通用技术之一。

## 1. 为什么 Redis 重要

Redis 在系统设计面试里出现频率很高，原因不是它“高级”，而是它非常通用。

它既能做：

1. 缓存
2. 计数器
3. 排行榜
4. 去重集合
5. 发布订阅
6. Stream 队列

也就是说，学透 Redis 往往比机械记很多零散产品更划算。

## 2. Redis 的核心特性

原文强调了两个关键词：

1. 多功能
2. 简单

Redis 的很多数据结构都和编码题里熟悉的结构一一对应，因此在面试里很适合展开说明。

同时，它的执行模型相对直接，没有太多难以解释的复杂优化器行为，这让它在面试里更容易讲清楚。

## 3. Redis Basic

Redis 是一个以内存为核心的数据结构存储系统。

原文提到它的关键特点包括：

1. In-Memory
2. 单线程执行模型
3. 对简单操作极快

这使它很适合承担高频读写、轻量计算和快速状态管理。

## 4. Redis 不是持久化数据库的直接替代品

Redis 的一个重要局限是持久性。

虽然可以通过 AOF 等方式降低数据丢失风险，但它通常不提供关系型数据库那种“事务提交即可靠落盘”的强持久保证。

因此：

1. 适合拿它做缓存、状态、计数、实时集合
2. 不适合把所有核心交易数据只放在 Redis

如果题目对 durability 要求非常高，Redis 往往只能做辅助层，而不是唯一主存储。

## 5. 常见数据结构

原文列出的基础数据结构包括：

1. Strings
2. Hashes
3. Lists
4. Sets
5. Sorted Sets
6. Bloom Filters
7. Geospatial Indexes
8. Time Series

在面试里，这些结构最常见的对应场景是：

1. String：缓存、计数器、简单 KV
2. Hash：对象缓存
3. Set：去重、成员判断
4. Sorted Set：排行榜、延迟队列、优先级场景
5. Stream：事件流、轻量消息处理

## 6. 通信模式

Redis 不只是缓存，也支持一些消息型能力，例如：

1. Pub/Sub
2. Streams

它们在某些简单场景下可以替代更重的系统，但不应把 Redis 误讲成 Kafka 的完全替代。

## 7. 命令模型

Redis 的命令非常直观，例如：

```text
SET foo 1
GET foo
INCR foo
XADD mystream * name Star surname Oconnor
```

这类命令式接口非常适合面试解释，因为：

1. 行为简单
2. 数据结构语义清晰
3. 容易推导复杂度和并发行为

## 8. 面试里怎么讲 Redis

如果题目里引入 Redis，建议先说明你打算把它用来做什么，而不是泛泛地说“加 Redis 提高性能”。

更好的表述是：

1. 用 Redis 缓存热点用户资料
2. 用 Sorted Set 做排行榜
3. 用 Set 做去重
4. 用 Stream 处理轻量异步任务

然后补两类风险：

1. 数据丢失 / 持久性不足
2. 热点 Key / 内存容量 / 故障回退

## 9. 结论

Redis 在系统设计面试里的价值，不只是“它快”，而是它能覆盖很多高频子问题，而且每一种用法都很好解释。

准备 Redis 时，重点不是背参数，而是掌握：

1. 各种数据结构分别解决什么问题
2. 哪些场景 Redis 合适
3. 哪些场景必须让持久化数据库兜底

## 补充内容

Notion 原文当前只覆盖了 Redis 基础，我补了一套更完整的 Redis 知识结构：

1. 核心定位：缓存、状态存储、轻量消息、计数和排行榜。
2. 数据结构：String、Hash、List、Set、Sorted Set、Stream。
3. 典型用法：缓存、分布式锁、限流、排行榜、去重、会话、延迟任务。
4. 持久化：RDB、AOF、混合持久化，各自的取舍。
5. 高可用：主从复制、哨兵、集群模式。
6. 风险点：热 key、大 key、内存淘汰、持久性不足、缓存击穿。
