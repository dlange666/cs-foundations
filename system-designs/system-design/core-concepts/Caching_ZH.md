# Caching

原文主题：缓存的使用时机、常见架构和面试表达方式。

## 1. 为什么要缓存

缓存几乎会出现在所有高读流量系统里，因为数据库常常先成为瓶颈。

核心收益有两个：

1. 降低读延迟
2. 降低数据库压力

典型对比是：数据库查询几十毫秒，而内存缓存读取可能只要 1 毫秒左右。

但缓存不是白赚，它也会引入失效、脏读、热点和故障回退问题。

## 2. 缓存放在哪里

缓存可以出现在多层：

1. Browser Cache
2. CDN
3. External Cache（如 Redis）
4. In-Process Cache

面试里最常提到的是：

1. 静态内容走 CDN
2. 应用热点数据走 Redis
3. 极热 Key 可以加本地进程缓存

## 3. 常见缓存架构

### 3.1 Cache-Aside

这是最常见、也最适合面试默认回答的模式。

流程：

1. 先查缓存
2. 命中则直接返回
3. 未命中则查数据库
4. 写回缓存
5. 再返回结果

优点：

1. 简单
2. 只缓存真正被访问的数据

缺点：

1. 首次未命中会有额外延迟

### 3.2 Write-Through

写请求先写缓存，再由缓存同步写数据库，只有两边都成功才算写入完成。

优点：

1. 缓存更容易和数据库保持一致

缺点：

1. 写延迟更高
2. 仍然存在双写问题

### 3.3 Write-Behind / Write-Back

应用先写缓存，缓存异步批量刷数据库。

优点：

1. 写入吞吐高

缺点：

1. 缓存挂掉时可能丢数据

因此更适合：

1. 指标
2. 分析
3. 可接受最终一致性的场景

### 3.4 Read-Through

应用不直接访问数据库，而是由缓存作为代理，在 miss 时回源数据库。

这个模式在业务系统里不如 cache-aside 常见，但 CDN 可以视为典型 read-through。

## 4. 驱逐策略

常见驱逐策略包括：

1. LRU
2. LFU
3. FIFO
4. TTL

面试里：

1. `LRU + TTL` 是很稳的默认组合
2. LRU 负责空间管理
3. TTL 负责防止数据无限陈旧

## 5. 常见缓存问题

### 5.1 Cache Stampede

热点缓存同时过期，大量请求一起回源数据库，导致数据库瞬时被打爆。

常见解决方案：

1. Request Coalescing / Single Flight
2. Cache Warming
3. 提前刷新热点 Key

其中最有效的通常是：只允许一个请求去重建缓存，其它请求等待结果。

### 5.2 Cache Consistency

最常见问题是缓存和数据库不一致。

常见策略：

1. 更新数据库后删除缓存
2. 用短 TTL 降低陈旧窗口
3. 对非关键读接受最终一致性

面试里要明确：不存在对所有系统都完美的一招，必须根据业务新鲜度要求做取舍。

### 5.3 Hot Keys

即使整体命中率很高，单个超热 Key 也可能压垮某个 Redis 节点或分片。

解决方式：

1. 热点 Key 复制
2. 本地进程缓存
3. 限流

## 6. 面试里什么时候引入缓存

不要上来就说“加 Redis”。先证明为什么需要缓存。

合适的触发点包括：

1. 读流量很大
2. 查询很贵
3. 数据库 CPU 很高
4. 延迟达不到 SLA

## 7. 面试里怎么讲缓存

推荐顺序：

1. 先指出瓶颈
2. 说明要缓存什么
3. 选择缓存模式
4. 指定缓存键设计
5. 说明驱逐策略
6. 补失效、一致性和故障回退

例如：

1. 用户画像和热点 Feed 是高频读且更新较少，适合缓存
2. 选 cache-aside
3. 用 Redis + TTL
4. 写操作后删除缓存
5. Redis 挂掉时回退数据库，并加熔断避免雪崩

## 8. 结论

缓存是数据库读路径太慢或太贵时的第一扩展手段。

它的核心取舍也很明确：

1. 用复杂度换性能
2. 用一定程度的数据陈旧换更低延迟和更高吞吐

面试里能把这两个点讲清楚，缓存部分通常就足够了。
