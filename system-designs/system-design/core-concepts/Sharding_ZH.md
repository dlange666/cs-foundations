# Sharding

原文主题：数据库分片的时机、策略和风险。

## 1. 什么是分片

当单机数据库在容量或吞吐上撑不住时，就需要把数据拆到多台机器上，这就是分片。

分片解决的是：

1. 存储容量上限
2. 写吞吐上限
3. 读吞吐上限

但分片会把系统从“单机数据库问题”升级为“分布式数据系统问题”。

## 2. Partitioning 和 Sharding 的区别

原文提醒这两个词常被混用。

更严格地说：

1. Partitioning：在单个数据库实例内拆分数据
2. Sharding：跨多台机器做水平拆分

面试里不用执着术语本身，但要讲清楚数据到底是在一台机器内拆，还是跨多台机器拆。

## 3. 常见分区方式

### Horizontal Partitioning

按行拆，把不同记录分到不同分区。

### Vertical Partitioning

按列拆，把不同字段分到不同表或存储里。

真正的 sharding 本质上是跨机器的 horizontal partitioning。

## 4. 如何选择 Shard Key

好的 shard key 应满足三个条件：

1. 高基数
2. 分布均匀
3. 与主查询模式对齐

如果这三个条件不满足，系统后面很容易出现热点、跨分片查询和重分片困难。

## 5. 分片策略

原文提到三类常见策略：

1. Range-Based Sharding
2. Hash-Based Sharding
3. Directory-Based Sharding

面试中最常用的默认思路通常是 hash-based，因为它更容易获得均匀分布。

Range-based 在时间范围查询里看起来很自然，但容易造成新数据写入集中到最新范围，形成热点。

Directory-based 更灵活，但会引入额外的路由元数据维护。

## 6. 分片后的典型问题

### 6.1 Hot Spots

即使 shard key 设计不错，仍然可能因为业务热点造成分片不均。

典型例子：

1. 某个明星用户流量远高于普通用户
2. 按时间分片导致最新分片承担全部新写入

常见解决办法：

1. 隔离热点 Key
2. 使用复合分片键
3. 动态拆分热点分片

### 6.2 Cross-Shard Queries

如果查询模式和 shard key 不一致，就会产生跨分片查询。

例如：

1. 你按 `user_id` 分片
2. 但要查“全站最热门的 10 条帖子”
3. 这就需要打到所有分片聚合

代价是：

1. 网络开销更高
2. 聚合逻辑更复杂
3. 尾延迟更差

缓解方式：

1. 缓存结果
2. 反规范化
3. 接受少量低频查询的高成本

### 6.3 Maintaining Consistency

单库里事务很自然；分片后，跨分片事务会明显复杂化。

理论上可以用 2PC，但在生产系统里它通常太慢、太脆弱。

因此更常见的做法是：

1. 设计上尽量避免跨分片事务
2. 用 Saga 等补偿思路处理多步骤操作
3. 接受部分链路的最终一致性

如果一个系统经常依赖分布式事务，通常意味着 shard boundary 或数据边界设计得不好。

## 7. 现代数据库里的分片

很多现代分布式数据库已经把分片和重平衡封装起来了。

例如：

1. Cassandra
2. DynamoDB
3. MongoDB

它们都能帮你做：

1. 路由
2. 分布
3. 扩容后的重平衡

但背后的机制并不完全相同，所以面试里不要笼统地说“都用一致性哈希”，要按技术本身解释。

## 8. 面试里什么时候提分片

不要过早分片。

只有当你已经明确单库在以下方面遇到瓶颈时，再提出分片：

1. 存储
2. 写吞吐
3. 读吞吐

推荐表达顺序：

1. 先指出瓶颈
2. 解释为什么单库不够
3. 选择 shard key
4. 选择分布策略
5. 说明热点、跨分片查询和未来扩容怎么处理

## 9. 结论

分片是数据库扩展的核心工具，但它不是免费的。

真正重要的只有两件事：

1. 分片键是否和访问模式匹配
2. 数据分布是否足够均匀

这两点如果做错，后面几乎所有问题都会被放大。

## 补充内容

Notion 原文已经覆盖了 shard key 和热点问题，我补了一套更适合工程落地的分片结构：

1. 触发条件：存储瓶颈、写吞吐瓶颈、读吞吐瓶颈。
2. 数据边界：按用户、租户、区域、时间还是业务对象分片。
3. 路由机制：客户端路由、代理路由、目录服务路由。
4. 负载问题：热点键、热点分片、写倾斜。
5. 一致性问题：跨分片事务、补偿、事件驱动同步。
6. 运维问题：重分片、迁移、扩容、回滚。
