# ElasticSearch

原文主题：ElasticSearch 的核心概念、基础使用方式，以及它在系统设计里的典型定位。

## 1. ElasticSearch 是什么

ElasticSearch 本质上是一个面向搜索场景的分布式文档系统。

你可以把它理解成：

1. 数据以文档形式存储
2. 用倒排索引做全文搜索
3. 擅长复杂检索、过滤、排序和地理位置查询

它不是传统事务数据库的直接替代品，更像是“搜索和检索层”。

## 2. 基础概念

原文先强调了四个最重要的客户端视角概念：

1. Document
2. Index
3. Mapping
4. Field

### Document

Document 是 ES 中最基本的数据单元，本质上就是一个 JSON 对象。

例如一本书可以表示为：

```json
{
  "id": "XX1123",
  "title": "The Great Gatsby",
  "author": "F. Scott Fitzgerald"
}
```

### Index

Index 是一组文档的集合。

可以把它近似理解为：

1. 在业务语义上像一张逻辑表
2. 在搜索系统里是一次查询的主要作用范围

原文也特别提醒：这里的 `index` 和数据库里的“索引结构”不是一个概念，容易混淆。

### Mapping 和 Field

Mapping 可以理解为索引的 schema，定义了：

1. 有哪些字段
2. 每个字段的类型
3. 字段如何被分析、索引和查询

例如：

```json
{
  "properties": {
    "id": { "type": "keyword" },
    "title": { "type": "text" },
    "author": { "type": "text" }
  }
}
```

这里最关键的取舍是：

1. `keyword` 适合精确匹配、过滤、排序
2. `text` 适合全文搜索

## 3. 为什么 Mapping 很重要

Mapping 不只是“字段说明”，它直接影响：

1. 哪些字段能搜
2. 搜索语义是什么
3. 集群内存和性能成本

原文举了一个很实用的点：

如果一个对象有 10 个字段，但你真正只按 2 个字段搜索，却把 10 个字段都映射并索引了，那就是纯粹的内存浪费。

所以在系统设计里，一个常见原则是：

`只索引真正需要搜索、过滤、排序的字段。`

## 4. 基础使用方式

### 创建索引

ES 通常通过 REST API 使用。

创建索引时可以指定：

1. 分片数
2. 副本数

例如：

```json
// PUT /books
{
  "setting": {
    "number_of_shards": 1,
    "number_of_replicas": 1
  }
}
```

### 设置 Mapping

如果不想完全依赖动态映射，就应该预先定义好字段类型。

这样可以：

1. 提前约束数据类型
2. 明确搜索语义
3. 避免动态映射带来的错误或膨胀

### 新增文档

向 `/_doc` 写入文档即可。

本质上就是：

1. 把 JSON 送进 ES
2. ES 按 Mapping 解析字段
3. 构建底层索引结构

## 5. 更新文档与并发控制

原文对更新这块讲得比较实用。

直接覆盖更新虽然简单，但在并发更新时可能互相覆盖。

因此你可以用版本号做乐观并发控制：

1. 更新时带上 version
2. 只有版本匹配才允许更新
3. 不匹配则返回错误，让客户端重试或合并

这就是典型的 optimistic concurrency control。

此外，`_update` 接口支持只更新部分字段，不必整份文档重写。

## 6. 搜索能力

### 基础搜索

最基础的查询方式就是 `match`。

例如按标题搜索包含 `Great` 的书：

```json
// GET /books/_search
{
  "query": {
    "match": {
      "title": "Great"
    }
  }
}
```

这类能力是 ES 相比普通数据库最核心的优势之一。

## 7. 地理位置搜索

原文专门补充了 geospatial search，这在很多系统设计题里非常有价值。

ES 支持两种常见地理字段：

1. `geo_point`：单个经纬度点
2. `geo_shape`：多边形、线、覆盖区域等复杂几何对象

适合场景：

1. Yelp
2. Uber
3. 外卖配送
4. 同城服务

常见查询是 `geo_distance`，例如查询 5km 内的餐厅。

原文还提到，在底层 ES 会使用 BKD tree 等结构来加速地理查询。

面试里不用深讲底层实现，但可以明确说：

`ES 原生支持地理位置索引和范围搜索，比普通关系型数据库更适合这类检索场景。`

## 8. 排序

ES 不只是“搜出来”，还要“排得对”。

常见排序方式包括：

1. 基础字段排序
2. 脚本排序
3. 嵌套字段排序
4. 按相关性排序

默认情况下，如果不指定 sort，ES 会按 `_score` 排序，也就是按相关性排序。

## 9. 分页

原文对分页讲得很完整，这也是系统设计题里常见追问。

### from / size

这是最直接的分页方式。

优点：

1. 简单
2. 易用

缺点：

1. 深分页性能很差
2. ES 需要跳过和排序大量前置结果

### search_after

这是一种更适合深分页的方式。

做法是：

1. 记录上一页最后一条结果的 sort 值
2. 下一页通过 `search_after` 从该位置继续向后取

优点：

1. 比 deep offset 高效
2. 不需要每次重扫所有前置结果

缺点：

1. 不能随机跳页
2. 客户端需要保留上一页状态

### PIT + search_after

如果你还希望分页期间的数据视图稳定，可以使用：

1. Point In Time
2. 配合 `search_after`

这样可以在分页过程中保持一致视图，避免底层数据变化导致前后页错乱。

## 10. ElasticSearch 在系统设计里的定位

ES 最适合做：

1. 全文搜索
2. 多条件过滤
3. 排序和相关性排名
4. 地理位置搜索

它不适合做：

1. 强事务主库
2. 高频跨文档事务更新
3. 需要严格 ACID 的交易核心数据

所以更准确的表达通常是：

`主业务数据放主存储，ES 作为搜索索引层或检索副本。`

## 11. 结论

ElasticSearch 的价值不只是“能搜”，而是它能把：

1. 文本检索
2. 条件过滤
3. 相关性排序
4. 地理位置查询
5. 深分页能力

统一放在一个搜索系统里解决。

面试里提到 ES 时，最重要的不是背 API，而是说明：

1. 为什么普通数据库不够
2. 为什么需要独立搜索层
3. ES 在系统里承担什么边界清晰的职责

## 补充内容

Notion 原文重点讲了概念和 API，我补了一层更偏系统设计决策的结构：

1. 数据入口：主库写入后如何同步到 ES，常见是 CDC 或异步消费。
2. 索引设计：哪些字段是 `text`，哪些字段是 `keyword`。
3. 查询设计：搜索、过滤、排序、聚合是否混用。
4. 性能设计：分片数、字段数量、deep pagination、热索引控制。
5. 一致性设计：ES 往往是最终一致的搜索副本，而不是唯一真相来源。
6. 失败设计：索引重建、写入积压、mapping 变更和查询降级。

## 常见问题

1. ElasticSearch 最适合解决什么问题？
- 最适合全文搜索、复杂过滤、相关性排序和地理位置检索。
- 如果题目核心是“搜得准、筛得快、排得合理”，ES 很可能是候选方案。

2. 为什么不能直接拿 MySQL 代替 ElasticSearch 做所有搜索？
- MySQL 能做简单查询，但不擅长复杂全文检索和相关性排序。
- 当搜索成为核心能力时，独立搜索引擎通常更合适。

3. `text` 和 `keyword` 的区别是什么？
- `text` 用于分词后的全文搜索。
- `keyword` 用于精确匹配、过滤、排序和聚合。

4. Mapping 为什么会影响性能？
- 因为每个被索引字段都会占用资源。
- 字段越多、类型越复杂，索引和查询成本越高。

5. ES 里的 Index 更像什么？
- 在业务理解上更像一组同类文档的逻辑集合。
- 但不要把它和数据库里的普通索引结构混为一谈。

6. 为什么更新文档时要提乐观并发控制？
- 因为并发写入时可能互相覆盖。
- 带版本号更新能显式处理写冲突。

7. `from/size` 为什么不适合深分页？
- 因为页数越深，ES 需要跳过和排序的结果越多。
- 这会让查询代价迅速升高。

8. `search_after` 适合什么场景？
- 适合深分页和顺序向后翻页。
- 不适合随机跳页场景。

9. 为什么 ES 通常不是系统的唯一数据源？
- 因为它更像搜索副本，而不是强事务主库。
- 真正的业务真相通常还在主数据库里。

10. 面试里什么时候应该主动提出 ElasticSearch？
- 当需求里明确有全文搜索、复杂筛选、相关性排序或地理位置检索。
- 如果只是按主键查单条记录，就没必要上 ES。
